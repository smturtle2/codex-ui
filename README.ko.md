# Codex WebUI

[English](./README.md) | [한국어](./README.ko.md)

Codex WebUI는 Codex 워크플로우를 위한 로컬 우선 인터페이스로 다시 만들고 있는 프로젝트입니다.

이 저장소는 이전 구현을 억지로 이어 붙이지 않습니다. 더 작은 범위, 더 분명한 제품 형태, 더 정직한 문서를 기준으로 처음부터 다시 시작합니다. README도 과거에 있던 것을 설명하는 문서가 아니라, 지금 이 저장소가 무엇을 만들려는지 설명하는 문서로 다시 씁니다.

## 개요

Codex WebUI의 목표는 단순합니다. Codex를 일반적인 채팅 래퍼로 평탄화하지 않으면서도, 브라우저에서 현대적이고 실용적으로 사용할 수 있는 인터페이스를 만드는 것입니다.

이번 재구축은 결국 다음과 같은 제품으로 가는 것을 목표로 합니다.

- 브라우저 안에서 Codex 워크플로우를 이해하기 쉽게 보여주기
- 기본값을 로컬 우선으로 유지하기
- 단순한 채팅 기록 뒤에 상태를 숨기지 않기
- 넓지만 불안정한 표면 대신 작지만 동작하는 코어에서 시작하기

## 왜 이 프로젝트를 다시 만드는가

Codex UI는 여전히 제대로 다시 설계할 여지가 있습니다.

- 일반적인 에이전트 채팅 셸은 중요한 상태와 워크플로우 경계를 쉽게 지워 버립니다.
- UI와 런타임 모델이 어긋나기 시작하면 로컬 도구는 금방 신뢰하기 어려워집니다.
- 더 이상 맞지 않는 구조를 붙잡는 것보다, 깨끗하게 다시 시작하는 편이 더 생산적입니다.
- 작은 코어부터 시작해야 아키텍처, 테스트, 문서 기준을 제대로 세울 수 있습니다.

## 재구축 방향

다음 버전은 몇 가지 분명한 축을 중심으로 설계합니다.

### 1. 플레이스홀더 데모가 아닌 실제 앱 셸

첫 번째로 쓸 수 있는 버전은 최소 기능이어도 일관된 앱 셸이어야 합니다. 처음부터 넓을 필요는 없습니다. 동작이 명확하고, 상태를 이해할 수 있고, 다시 전부 갈아엎지 않고 확장할 수 있으면 됩니다.

### 2. UI와 Codex 런타임 사이의 분명한 경계

브라우저 레이어, 로컬 애플리케이션 레이어, Codex 연동 레이어는 각자 책임이 분명해야 합니다. 이번 재구축에서는 그 경계가 한 덩어리 UI 코드로 무너지지 않도록 설계합니다.

### 3. 기본값은 로컬 우선

이 프로젝트는 먼저 로컬 사용성을 위해 만듭니다. 원격 배포, 멀티유저, 플랫폼 확장 요구는 실제 필요가 드러나기 전까지 초기 구조를 흔들지 않아야 합니다.

### 4. 바깥에서 이해 가능한 제품 동작

중요한 상태, 전이, 사용자 액션은 바깥에서 읽히고 디버깅 가능해야 합니다. 제품이 어떤 식으로 동작하는지 추측하게 만들지 않는 것이 목표입니다.

## 현재 상태

현재 저장소는 의도적으로 리셋된 상태입니다.

- 이전 구현은 정리되었습니다.
- 새로운 실행 가능한 기준선은 아직 체크인되지 않았습니다.
- 실제 실행 대상이 생기기 전까지 Setup과 Quick Start는 넣지 않습니다.
- 기술 선택, 디렉터리 구조, 내부 경계는 처음 원칙부터 다시 정리하고 있습니다.

## 초기 빌드 범위

재구축은 방향을 증명할 수 있는 가장 작은 버전부터 시작합니다.

- 깔끔한 프로젝트 스캐폴드 정리
- 최소 앱 셸 구축
- Codex 연동 경계 정의
- 핵심 대화 흐름 복원
- 실제 구현과 함께 테스트와 운영 문서 다시 추가

## 원칙

- 문서는 항상 코드베이스와 맞아야 합니다.
- 영리한 우회보다 적은 구성 요소를 택합니다.
- 상상 속 미래 유연성보다 실제 워크플로우를 기준으로 설계합니다.
- 넓어지기 전에 코어를 먼저 유용하게 만듭니다.
- 리셋은 혼란을 이름만 바꾸는 기회가 아니라, 실제로 걷어내는 기회로 사용합니다.

## 이 README의 역할

이 README는 재구축 방향을 설명하는 문서입니다.

- 새 버전의 프로젝트가 어디로 가는지 설명합니다.
- 이미 삭제된 구현의 제품 매뉴얼이 아닙니다.
- 아직 Quick Start 문서는 아닙니다.
- 새 기준선이 생기면 다시 운영 문서 성격을 더해 갈 예정입니다.
